# README Pattern Research

Use this only when studying README examples or updating `$mydoc-readme` defaults. Do not load it during ordinary README generation.

Source set: `references/repo-readmes-index.md`, copied from `/Users/jordanmamroud/mylab/repos-study`.

## Adopt

- Lead with a plain product/project explanation, then show the shortest credible path to use it.
- Add `Capabilities` when the repo exposes tools, skills, services, commands, APIs, or user-facing workflows.
- Add `How It Works` when the repo has a distinct workflow, architecture, or mental model that a new reader must understand before using it.
- For jobs, automations, CLIs, services, and pipelines, make `How It Works` a concrete numbered flow from trigger to outcome, naming the components, artifacts, safety gates, and reporting steps that repo evidence supports.
- Use bold action titles in each numbered step: `1. **Runtime config is loaded.** node run.mjs reads...`
- Add `Requirements` before setup when use depends on runtimes, CLIs, auth, external accounts, credentials, or platform-specific installs.
- Add `Configuration` when env vars, config files, credentials, OAuth, or client setup are required.
- Keep `Quick Start` executable and short; move long install/auth/deploy details into `Requirements`, `Configuration`, `Common Tasks`, or deeper docs.
- Add `Support` only when issue trackers, troubleshooting docs, support channels, or known failure-mode docs exist.
- Add `Contributing` and `License` only when the repo has explicit files or metadata for them.

## Avoid As Defaults

- Badges, star charts, sponsorship sections, language switchers, and community links are useful for public open-source projects but should not be generated by default.
- Long setup guides should not bury the first command. Keep the root README concise and link to deeper docs when available.
- Do not compress an operational workflow into vague summary bullets when the repo has a clear step-by-step runtime flow.
- Do not invent maturity, support, licensing, production status, auth scopes, deploy targets, or command behavior.

## Best Default Order

1. Title
2. Status
3. Summary
4. Capabilities
5. How It Works
6. Requirements
7. Quick Start
8. Configuration
9. Common Tasks
10. What's Here
11. Related Docs
12. Support
13. Contributing
14. License
